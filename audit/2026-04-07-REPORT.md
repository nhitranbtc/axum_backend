## 🔴 Critical Findings

- **[CRITICAL]** JWT_SECRET has a hardcoded insecure fallback — if env var is missing,
  the app uses `"dev-secret-change-in-production"` silently instead of failing.
  → `src/config/app_config.rs:32`

- **[CRITICAL]** Cookie `secure` flag hardcoded to `false` — tokens will be sent over
  plain HTTP in production. Comments say "Set to true in production" but code
  never reads config to determine this.
  → `src/presentation/handlers/auth.rs:121,129`

- **[CRITICAL]** Refresh tokens stored UNHASHED — `RefreshToken::new()` receives the
  raw JWT string; the field is named `token_hash` but no hashing occurs. A DB
  breach exposes live refresh tokens directly.
  → `src/application/use_cases/auth/login.rs:108-113`
  → `src/domain/entities/refresh_token.rs:16`

- **[CRITICAL]** Confirmation codes generated with `rand::thread_rng()` (PRNG), not
  `OsRng` (CSPRNG). Codes are also only 6 digits (1-in-1,000,000 brute-force).
  → `src/application/use_cases/auth/register.rs:86`
  → `src/application/use_cases/auth/forgot_password.rs:51`
  → `src/application/use_cases/auth/resend_code.rs:59`

- **[CRITICAL]** Blocking I/O inside async handler — `std::fs::read()` called directly
  in an async function, blocking the Tokio thread pool.
  → `src/presentation/handlers/user.rs:42`

- **[CRITICAL]** Argon2 hashing called without `spawn_blocking` in the import path —
  CPU-heavy hashing on the async executor thread.
  → `src/application/use_cases/user/import.rs:48`

- **[CRITICAL]** No rate limiting on any auth endpoint — register, login, forgot-password,
  and resend-code are all unprotected against brute-force and enumeration attacks.
  → `src/presentation/routes/auth.rs` (no tower_governor or similar middleware)

## 🟡 Warnings

- **[WARN]** Swagger UI (`/swagger-ui`) and Prometheus `/metrics` endpoint are public with
  no authentication gate. In production, this leaks API schema and system metrics.
  → `src/presentation/routes/mod.rs:160,162`

- **[WARN]** No RBAC enforcement in privileged handlers — `create_user`, `update_user`,
  `update_user_role`, and `import_users` check authentication (JWT present) but
  do not verify the caller's role. Any authenticated user can create users or
  escalate roles.
  → `src/presentation/handlers/user.rs:84,162`
  → `src/presentation/handlers/role.rs:64`

- **[WARN]** `panic!()` in `JwtManager::new()` — constructor panics on short secret
  instead of returning Result, violating the "constructors return Result" rule.
  → `src/shared/utils/jwt.rs:47`

- **[WARN]** `panic!()` inside `user_model_to_entity` — if an invalid email somehow
  reaches the DB, this panics the request thread.
  → `src/infrastructure/database/repositories/auth.rs:37`

- **[WARN]** `Email::parse(email).unwrap()` in `create_user` on the DB-returned email —
  same panic risk in a hot code path.
  → `src/infrastructure/database/repositories/auth.rs:144`

- **[WARN]** `std::sync::Mutex::lock().unwrap()` in async monitoring handler — should
  use `tokio::sync::Mutex` in async context.
  → `src/infrastructure/monitoring.rs:26`

- **[WARN]** `reqwest` appears in both `[dependencies]` and `[dev-dependencies]` in
  `Cargo.toml` (lines 86 and 105). If not used in production code, it should be
  dev-only — adds unnecessary binary bloat and attack surface.
  → `Cargo.toml:86`

- **[WARN]** Stale/broken unit tests — `user.rs` tests call `User::new(email, name,
  password_hash)` with 3 args, but the current signature only accepts 2
  (`email, name`). These tests cannot compile.
  → `src/domain/entities/user.rs:149-181`

- **[WARN]** Import actor spawns one actor per CSV row — for large imports, this
  creates unbounded goroutines-equivalent processes and is a DoS vector if the
  CSV is attacker-controlled. No limit on actor count.
  → `src/application/use_cases/user/import.rs:52-55`

- **[WARN]** `main.rs` uses `.expect("Failed to create email service")` — panics at
  startup rather than returning a typed error via `main() -> Result<>`.
  → `src/main.rs:33`

- **[WARN]** CQRS migration incomplete — `application/use_cases/` and
  `application/commands/` + `application/queries/` coexist. New auth flows were
  added to the legacy `use_cases/` path.
  → `src/application/use_cases/auth/`

## 🟢 Strengths

- JWT validation is strict: `exp`, `sub`, `iat`, `jti`, `iss`, `aud` all required and validated,
  `token_type` checked; access vs refresh token differentiation enforced.
  → `src/shared/utils/jwt.rs:101-119`

- Auth middleware correctly passes claims via request extensions; handlers use
  `Claims` as an Axum extractor with a clean rejection path.
  → `src/presentation/middleware/auth.rs`

- Repository trait pattern is clean: domain traits with concrete impls in
  infrastructure; Diesel queries use `.optional()` correctly and all parameterized.
  → `src/infrastructure/database/repositories/`

- Error types are well-structured with `thiserror`; each use case defines its own
  typed error enum; INTERNAL errors are logged with `tracing::error!` and only
  generic messages returned to clients.
  → `src/application/use_cases/auth/`

- DDD layer separation is largely respected: domain has zero infrastructure imports;
  value objects (`Email`, `UserId`, `UserRole`) enforce invariants at construction.
  → `src/domain/`

- Password hashing uses Argon2 with OsRng and is wrapped in `spawn_blocking` in the
  `set_password` use case.
  → `src/application/use_cases/auth/set_password.rs:68`
  → `src/shared/utils/password.rs:24`

- Good test infrastructure: `testcontainers` for ephemeral PostgreSQL, `axum-test`
  for handler-level integration, `mockall` for trait mocking, Criterion benchmarks.
  → `tests/common/`

- Swagger/OpenAPI annotations are complete and schema-consistent.
  → `src/presentation/routes/mod.rs:39-98`

## Dimension Breakdown

### 1. Structure (7.5 / 10)

The directory layout cleanly maps to DDD layers: `domain/`, `application/`, `infrastructure/`,
`presentation/`, `shared/`, `config/`. Module naming is consistent. The 108-file count is
well-organized and navigable. The only structural friction is the parallel CQRS
migration leaving `use_cases/`, `commands/`, and `queries/` all active simultaneously,
and the `import_users` handler reaching into `std::fs` — a domain concern leaking
into the HTTP layer.

**Issues:**
- Dual `use_cases/` and `commands/`+`queries/` directories during migration → `src/application/`
- File I/O in presentation handler (should be in a service/use case) → `src/presentation/handlers/user.rs:42`

**Recommendations:**
- Complete the CQRS migration: migrate remaining `use_cases/auth/` to `commands/` + `queries/`
- Move CSV file reading into a dedicated `FileImportService` in infrastructure; handler receives bytes only

---

### 2. Architecture & Design (6.5 / 10)

The repository-trait + `Arc<dyn Trait>` pattern is solid. Use cases are thin and delegate
correctly. The Actor pattern for bulk import is creative but problematic at scale.
`create_router` takes 10 positional parameters — a builder or `RouterConfig` struct
would improve ergonomics significantly. `ResendConfirmCodeUseCase` is constructed inline
in the router rather than alongside its siblings.

**Issues:**
- `create_router` 10-parameter signature → `src/presentation/routes/mod.rs:116`
- Unbounded actor-per-row in import → `src/application/use_cases/user/import.rs:52`
- No graceful shutdown wired (known tech debt)
- `UserService::get_user_count()` returns hardcoded `0` (known tech debt)

**Recommendations:**
- Introduce a `RouterDeps` or builder struct for router construction
- Limit import concurrency with a bounded semaphore (e.g., `tokio::sync::Semaphore`)
- Wire graceful shutdown with `tokio::signal`

---

### 3. Security (3.5 / 10)

Multiple critical gaps exist that would block production deployment. The hardcoded
JWT fallback, unhashed refresh tokens, cookie insecurity, and lack of rate limiting
are the most severe. RBAC is architecturally modeled (`UserRole` value object exists)
but never enforced at the handler layer. Confirmation codes are cryptographically weak.

**Issues:**
- Hardcoded JWT_SECRET fallback → `src/config/app_config.rs:32`
- Unhashed refresh tokens in DB → `src/application/use_cases/auth/login.rs:108`
- Cookie `secure=false` hardcoded → `src/presentation/handlers/auth.rs:121,129`
- `thread_rng` for confirmation codes (not CSPRNG) → `src/application/use_cases/auth/register.rs:86`
- No rate limiting on auth routes
- Swagger UI and `/metrics` public → `src/presentation/routes/mod.rs:160,162`
- No RBAC checks in privileged handlers → `src/presentation/handlers/user.rs:84`

**Recommendations:**
- Remove JWT_SECRET fallback; use `.map_err(|_| ConfigError::MissingEnvVar(...))?`
- Hash refresh tokens with SHA-256 before storing; compare hashes on lookup
- Drive `secure` flag from `AppConfig.is_production` bool
- Replace `rand::thread_rng()` with `OsRng` for confirmation codes; use 32-byte hex
- Add `tower_governor` or custom rate-limit middleware on `/api/auth/*`
- Gate Swagger behind feature flag (already done) + auth middleware in production
- Add RBAC check at the top of privileged handlers: check `claims.role == UserRole::Admin`

### 4. Code Quality (6.0 / 10)

Error handling follows the project's own conventions in most places, but there are
notable violations: two `panic!()` calls in production paths, several `.unwrap()` calls
outside of test modules, and stale test code that mismatches the current API. The
overall code is readable with consistent formatting, but TODO-style comments remain
in production files (e.g., `login.rs` and `register.rs` have multi-line comment blocks
describing design decisions inline).

**Issues:**
- `panic!()` in `JwtManager::new` → `src/shared/utils/jwt.rs:47`
- `panic!()` in `user_model_to_entity` → `src/infrastructure/database/repositories/auth.rs:37`
- `.unwrap()` on `Email::parse` in production → `src/infrastructure/database/repositories/auth.rs:144`
- Stale tests with wrong `User::new` signature → `src/domain/entities/user.rs:149-181`
- Design-discussion comments in production code → `src/application/use_cases/auth/login.rs:41-50`

**Recommendations:**
- Convert `JwtManager::new` to return `Result<Self, JwtError>`
- Convert `user_model_to_entity` to return `Result<User, RepositoryError>`
- Clean up inline TODO-style design comments before production

---

### 5. Testing (6.0 / 10)

The test infrastructure is genuinely strong: ephemeral containers, handler-level tests,
mocking, benchmarks. But the stale unit tests in `user.rs` will fail to compile (wrong
arity on `User::new`), meaning the test suite may be partially broken. CI is configured
(`.github/workflows` present) which is positive. No evidence of property-based testing
or fuzz testing for value objects.

**Issues:**
- Stale tests (`User::new` arity mismatch) likely cause compilation failures → `src/domain/entities/user.rs`
- No tests for confirmation code expiry edge cases
- No tests for RBAC enforcement (because RBAC is not implemented)
- Coverage depth for error paths is unclear

**Recommendations:**
- Fix or delete stale unit tests immediately
- Add tests for: expired confirmation codes, rate limit behavior, token replay prevention
- Add property tests for `Email` and `UserRole` value objects

---

### 6. Dependencies (7.0 / 10)

Core dependency choices are sound: Diesel 2.1 with diesel-async for async Postgres,
Argon2 for password hashing, jsonwebtoken 9.0 with strict validation, utoipa for
type-safe OpenAPI. `Cargo.lock` is committed (correct for a binary). The main issues
are `reqwest` in production deps and `rand` 0.8 used for PRNG-based code generation where
CSPRNG is required.

**Issues:**
- `reqwest` in `[dependencies]` (should be dev-only) → `Cargo.toml:86`
- `rand::thread_rng()` used for security-sensitive confirmation codes
- No `cargo audit` or `cargo deny` configuration visible

**Recommendations:**
- Move `reqwest` to `[dev-dependencies]` if not used in production binary
- Replace `rand::thread_rng()` with `rand::rngs::OsRng` for confirmation codes
- Add `deny.toml` (cargo-deny) for supply-chain and license checks in CI

### 7. Documentation (7.5 / 10)

`CLAUDE.md` is exceptional — thorough, opinionated, and accurately describes known tech
debt. The README is present. Swagger UI provides live API documentation. `MEMORY.md`
supplements `CLAUDE.md`. The gap is no `CHANGELOG.md` and minimal inline doc comments
on public traits/structs in the domain layer.

**Issues:**
- No `CHANGELOG.md`
- Domain repository traits lack doc comments → `src/domain/repositories/`
- No runbook or deployment documentation

**Recommendations:**
- Add `CHANGELOG.md` (even sparse, start with current version)
- Add `///` doc comments to repository traits for IDE discoverability

## Verdict

This codebase has a solid DDD foundation, clean layer separation, and thoughtful
infrastructure choices (Diesel async, Argon2, strict JWT validation). However, it
is **not production-ready** in its current state due to multiple critical security
gaps: an insecure JWT secret fallback, unhashed refresh tokens, cryptographically
weak confirmation codes, hardcoded insecure cookie flags, zero rate limiting, and
no RBAC enforcement despite the role model being defined. The architecture is
heading in the right direction, but the security layer needs a focused hardening
pass before this service should handle real users.
